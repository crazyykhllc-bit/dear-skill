# 增量合并逻辑

## 任务
用户追加新的原材料后，分析增量内容，merge 进现有的 persona.md / care.md / memory.md。

---

## 合并原则

### 1. 不覆盖已有结论
- 新信息是**追加**，不是**替换**
- 如果新信息与已有结论矛盾，标注为 `[新信息与已有记录冲突，需用户确认]`

### 2. 分流规则
分析增量内容属于哪个文件：

| 内容类型 | 归入文件 |
|---------|---------|
| 说话方式、口头禅、性格表现 | persona.md |
| 关心行为、爱的表达、唠叨内容 | care.md |
| 具体事件、地点、食物、故事 | memory.md |
| 情绪反应、价值观表达 | persona.md (Layer 3) |
| 经典操作、矛盾行为 | care.md |

### 3. 合并格式
在对应文件的对应节追加内容，用以下标记：

```markdown
<!-- 增量更新 {日期} -->
{新增内容}
```

### 4. 质量提升
如果新的原材料质量更高（如从"口述"升级到"聊天记录"），可以：
- 用更精确的描述替换之前的推断
- 将 `[信息不足，推断]` 标记改为确认

### 5. SKILL.md 重新生成
合并完成后，重新合并三个文件生成 `SKILL.md`。

---

## 执行流程

1. 读取已有的 persona.md、care.md、memory.md
2. 分析新原材料
3. 判断新内容属于哪个文件的哪个章节
4. 追加到对应位置
5. 如有冲突，提示用户
6. 更新 meta.json 的 version 和 updated_at
7. 重新生成 SKILL.md
